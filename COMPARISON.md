# IA vs. Human Comparison / Comparativa: IA vs. Humano

This document analyzes the two implementations in this project: the `ia` package (generated by the AI assistant) and the `human` package (historical implementation by the user).

---

## 🇪🇸 Castellano

### 1. Arquitectura y Diseño

| Característica | Solución IA (`es.aam.kata.ia`) | Solución Humana (`es.aam.kata.human`) |
| :--- | :--- | :--- |
| **Estructura** | Una sola clase concisa y directa. | 3 clases (Fachada + 2 Lógicas separadas). |
| **Enfoque** | Orientado a Datos (Data-Driven). | Orientado a Procedimientos y Reglas. |
| **Complejidad** | Baja. Fácil de entender de un vistazo. | Alta. Fragmentada en múltiples métodos. |

### 2. Análisis del Algoritmo

*   **Decimal a Romano:**
    *   **IA**: Enfoque "greedy" con arrays de valores. Eficiente y estándar.
    *   **Humana**: Descompone el número mediante `Math.pow` y `String.valueOf`. Es ingenioso pero innecesariamente costoso.
*   **Romano a Decimal:**
    *   **IA**: Procesa de derecha a izquierda, simplificando la lógica de la resta.
    *   **Humana**: Busca "números compuestos" (IV, IX) con `substring`, lo que requiere mayor control de índices.

### 3. Validación Funcional
Gracias al `RomanComparisonTest`, hemos verificado que **ambas soluciones son 100% equivalentes** en el rango del 1 al 3999. No hay discrepancias en los resultados, solo en el "cómo" se llega a ellos.

### 4. Conclusión
La **solución humana** demuestra un pensamiento algorítmico basado en la descomposición manual (como lo haría una persona con papel y lápiz), mientras que la **solución IA** optimiza para el procesamiento computacional utilizando estructuras de datos.

## 🇺🇸 English

### 1. Architecture and Design

| Feature | AI Solution (`es.aam.kata.ia`) | Human Solution (`es.aam.kata.human`) |
| :--- | :--- | :--- |
| **Structure** | Single concise class. | 3 classes (Facade + 2 separate logics). |
| **Approach** | Data-Driven. | Procedural / Rule-based. |
| **Complexity** | Low. Easy to grasp at a glance. | High. Fragmented into multiple methods. |

### 2. Algorithmic Analysis

*   **Decimal to Roman:**
    *   **AI**: Greedy approach using value arrays. Efficient and standard.
    *   **Human**: Decomposes numbers using `Math.pow` and `String.valueOf`. Clever but unnecessarily heavy.
*   **Roman to Decimal:**
    *   **AI**: Processes right-to-left, simplifying subtraction logic.
    *   **Human**: Scans for "composite numbers" (IV, IX) using `substring`, requiring more complex index management.

### 3. Functional Validation
Through the `RomanComparisonTest`, we have verified that **both solutions are 100% equivalent** for the range 1 to 3999. There are no discrepancies in output, only in the implementation strategy.

### 4. Conclusion
The **human solution** showcases algorithmic thinking based on manual decomposition (simulating how a person would do it with pen and paper), whereas the **AI solution** optimizes for computational processing by leveraging efficient data structures.
